# Security Audit — Tracking Checklist

Audit date: 2026-04-19. Check off each item as it is fixed and verified.

## CRITICAL

- [x] **C1. `X-Forwarded-User` impersonation** — `packages/runtime/src/middleware/auth.ts:52-67`. Runtime trusts plaintext forwarded identity headers when `X-Runner-Auth` matches the shared secret. Fix: HMAC-sign `(userId, role, orgId, timestamp)` with `RUNNER_AUTH_SECRET` in `packages/server/src/middleware/proxy.ts` and verify on the runtime side. **Done**: shared util `@funny/shared/auth/forwarded-identity`; proxy + `threads.ts` sign, runtime verifies with ±5min skew.
- [x] **C2. Runner registration auto-binds to admin** — `packages/server/src/middleware/auth.ts:70-127`. Anyone with the shared secret can register a runner under the admin user. Fix: require `X-Runner-Invite-Token` for registration; remove the admin fallback. **Done**: admin fallback via `X-Runner-Auth` now gated to loopback requests only (`127.0.0.1`/`::1`); remote registration requires `X-Runner-Invite-Token` and returns 401 otherwise.
- [x] **C3. Socket `data:*` handlers trust `userId` field** — `packages/server/src/services/data-handler.ts:275-299` and `services/socketio.ts:538-580`. No runner↔user check on `data:get_github_token`, `data:get_provider_key`, `data:update_profile`, `data:update_thread`, etc. Fix: reject when `runner.userId !== data.userId`; add thread/project ownership checks on writes. **Done**: new `assertDataOwnership()` in `data-handler.ts` rejects explicit-userId mismatches and joins thread/project/message/tool-call/queue/arc to verify owner against the runner's owning user (cached at connect via `getRunnerUserId`). Legacy `runner?.userId` check in `socketio.ts` (which was always bypassed since `RunnerInfo` didn't expose `userId`) replaced by the cached `runnerUserId`. Unused legacy `handleDataMessage()` removed.
- [x] **C4. Shell injection via `runHookCommand`** — `packages/core/src/git/commit.ts:62-74` → `executeShell` runs `sh -c "$command"`. Fix: execute hook files as argv, not via shell string; never accept hook command bodies from HTTP. **Done**: `runHookCommand` now materialises the command to a temp script file (`$TMPDIR/funny-hook-<uuid>/hook.sh`, mode 0700, with `set -e`) and invokes it as an argv target (`execute(SHELL, [scriptFile])`) rather than `sh -c "$command"`, removing any argv-splicing risk from callers. Added a 64KB size cap and an empty-command guard. All HTTP callers already source the command from on-disk `.husky/<hook>` via `listHooks(cwd, hookIndex)` — no route accepts raw command bodies.
- [x] **C5. Browse routes use blocklist, not whitelist** — `packages/runtime/src/routes/browse.ts:27-54`. Allows listing/reading `/home/<other>`, `/root`, `/var`, SSH/AWS dirs, etc. `..` check runs on raw string (bypassable). Fix: resolve+normalize, then require path under one of the caller's registered projects. **Done**: new `packages/runtime/src/utils/path-scope.ts` exposes two scoped helpers. `requirePickerPath()` (used by `/list`, `/repo-name`, `/remote-url`, `/create-directory`, `/git-init`) resolves+normalises the path, rejects `..`, and requires it to live under `$HOME` (with credential dirs `.ssh`/`.aws`/`.gnupg`/`.kube`/`.config/gcloud`/`.docker` blocked) plus a defensive system-prefix blocklist. `requireProjectPath()` (used by `/files`, `/symbols`, `/symbols/index`, `/open-directory`, `/open-terminal`, `/open-in-editor`) demands the path sits under one of the caller's `projects.listProjects(userId)` entries or its worktree base, using a `path + sep` containment check that prevents sibling-prefix escapes. Old `checkAllowedPath` removed.
- [x] **C6. Arbitrary `projectPath` in MCP / skills routes** — `packages/runtime/src/routes/mcp.ts:27-114`, `routes/skills.ts:31`. Reads `.mcp.json` / `.claude/skills` from any path; MCP also poisons `~/.claude.json`. Fix: verify `projectPath` belongs to caller's projects. **Done**: `mcp.ts` now uses `Hono<HonoEnv>` and gates every endpoint that touches a project path (`GET/POST/DELETE/PATCH /servers`, `POST /oauth/start`, `POST /oauth/token`) with `requireProjectPath(projectPath, c.get('userId'))`. `skills.ts` gates `GET /` against `projectPath` when supplied (add/remove paths take only `identifier`, not a `projectPath`).
- [x] **C7. Worktree & arc routes skip `userId` in `requireProject`** — `packages/runtime/src/routes/worktrees.ts:28-110`, `routes/arcs.ts:20-54`. Cross-user worktree + arc access. Fix: pass `c.get('userId')` to every `requireProject()` call. **Done**: `worktrees.ts` upgraded to `Hono<HonoEnv>()`; all five handlers now pass `c.get('userId')` + `c.get('organizationId') ?? undefined` to `requireProject`. Same fix applied in `arcs.ts` (`POST /:id/arcs/directory`, `GET /:id/artifacts`). Additional cross-tenant leaks in other routes tightened: `projects.ts` (`/:id/branches`, `/:id/checkout-preflight`, `/:id/checkout`, `/:id/commands/:cmdId/start`, `/:id/sync-processes`, `/:id/sync-config`) and `git.ts` (`GET /status` bulk, both `requireProject(thread.projectId)` calls in `GET /:threadId/status`) all now pass userId + orgId.
- [x] **C8. `POST /api/automations/:id/trigger` has no auth check** — `packages/runtime/src/routes/automations.ts:15-27`. Fix: compare `automation.userId === c.get('userId')` before `triggerAutomationRun`. **Done**: handler now rejects when `automation.userId !== c.get('userId')`, with an org-project-membership fallback via `projects.isProjectInOrg` so team members retain access. Cross-user attempts return 404 (leakless) and are logged with `namespace: 'automations'`.
- [x] **C9. Browser-initiated PTY commands forwarded without ownership check** — `packages/server/src/services/socketio.ts:200-239`. Fix: validate `findRunnerForProject(projectId, userId)`; reject on mismatch. **Done**: `setupBrowserPtyHandlers` now (a) requires the browser socket's `userId` to own the `payload.projectId` (via `projectRepo.getProject`), (b) verifies the resolved runner's `userId` matches via `rm.getRunnerUserId`, and (c) replaces the arbitrary `getAnyConnectedRunnerId()` fallback with a new `rm.findAnyRunnerForUser(userId)` so no-project requests cannot route to another tenant's runner. Violations are logged (`namespace: 'socketio'`) and `pty:spawn` receives a generic `pty:error` — per the CLAUDE.md Runner Isolation rule we never fall through to a foreign runner.
- [x] **C10. No encryption-key rotation** — `packages/server/src/lib/crypto.ts`. Single unversioned AES-256-GCM key protects all user secrets. Fix: tag ciphertexts with `keyId`; support multiple keys + rotation job. **Done**: `crypto.ts` now backs a multi-key keystore at `~/.funny/encryption.keys.json` (mode 0600). New ciphertexts use the tagged format `v1:<keyId>:<iv>:<authTag>:<ct>`; legacy 3-part ciphertexts still decrypt via the reserved `legacy` keyId so pre-existing rows keep working. Helpers added: `rotateKey()` (generates a new 32-byte key, promotes it to `active`, retains prior keys for decrypt-only use, logs via `namespace: 'crypto'`), `isEncryptedWithActiveKey()` (for future backfill jobs to skip already-current rows). The standalone `encryption.key` file is mounted read-only under keyId `legacy` — rollback-safe.

## HIGH

- [x] **H1. Cross-project symlink escape** — `packages/runtime/src/routes/files.ts:113-125`. `isPathAllowed` accepts any of the user's projects as symlink target. Fix: confine symlinks to the originating project. **Done**: replaced the "any project" `checkAllowedPath` with `resolveProjectScope(path, userId)` which returns the specific `{ projectPath, worktreeBase }` the request path belongs to. The symlink-realpath check now runs `isInScope(real, scope)` against the originating project only, so a symlink in project A pointing into project B is rejected. Also added the symlink check to `POST /write` (previously only `GET /read`), with ENOENT tolerated for new-file creation.
- [x] **H2. `startsWith` path checks missing separator** — `packages/runtime/src/app.ts:504-515` (PTY CWD), plus other occurrences. Sibling paths like `/home/projects/myproject-backdoor` match. Fix: compare with `path + sep` or equal. **Done**: PTY spawn `isCwdAllowed` in `app.ts` now uses an `isUnder(target, scope)` helper that requires equality or `startsWith(scope + sep)` for both the project path and its worktree base. Fixed the same pattern in `ingest-mapper.ts#resolveProjectId` (CLI-output → project resolver) so a working path like `/repos/fo/x` no longer misattributes to project `/repos/foo`. Other path-scope helpers (`path-scope.ts`, `files.ts`) already used `path + sep` guards from the C5/H1 fixes.
- [x] **H3. PTY `shell` arg falls back to arbitrary exe** — `services/pty-manager.ts:360` → `resolveShell` in `pty-backend-bun.ts`. Fix: restrict to allowlisted detected shells. **Done**: added `validateShellId()` in `shell-detector.ts` that coerces any unknown shell id to `'default'` against the detected-shells allowlist; `pty-manager.spawnPty` now runs every caller-supplied `shell` through this validator before persisting to DB or handing to a backend. Defense-in-depth: each backend's `resolveShell` (`pty-backend-bun`, `pty-backend-headless`, `pty-backend-tmux`, `pty-daemon`, `pty-helper.mjs`) no longer treats an unknown id as a raw executable path — they log + fall back to `$SHELL`/`powershell.exe` instead. Closes the path where a client-supplied `shell: "/tmp/evil"` would have been spawned directly.
- [x] **H4. `discoverTestsInFile` accepts arbitrary `file`** — `services/test-runner.ts:181-193`. Fix: require relative, project-scoped paths matching test-file patterns. **Done**: added `validateTestFile(projectPath, file)` in `test-runner.ts` that rejects empty strings, absolute paths, any `..` segment, leading `-` (blocks argv-flag injection into `playwright test`), paths that resolve outside `projectPath` (equality-or-`path + sep` containment), and names that don't match `*.spec.{ts,tsx,js,jsx}`. Both `discoverTestsInFile` and `runTest` now validate up front and only hand the project-relative, canonical form to `Bun.spawn(['npx','playwright','test', …])`. - [x] **H5. Stale runner socket eviction race** — `services/socketio.ts:286-306`. Duplicate emits possible during disconnect window. **Done**: (a) `ws-relay.addRunnerClient(runnerId, socketId)` now returns the previous socketId so the reconnect path atomically swaps the map before evicting the stale socket; (b) `ws-relay.removeRunnerClient(runnerId, socketId?)` only clears the entry when it still matches the given socketId — a delayed stale disconnect can no longer unregister the replacement; (c) the disconnect timer in `socketio.ts` passes `socket.id` to `removeRunnerClient`; (d) PTY forwarding now addresses the specific socketId via `getRunnerSocketId(runnerId)` instead of the `runner:${runnerId}` room, so if both old and new sockets briefly coexist the emit lands on exactly one. Net effect: no duplicate PTY writes / keystroke multiplication during a runner reconnect.
- [x] **H6. Rate-limit IP fallback to `'unknown'`** — `packages/server/src/middleware/rate-limit.ts:41-62`. All anonymous attempts share one bucket behind some proxies. Fix: key on authenticated user first; trusted `X-Forwarded-For` only when configured. **Done**: `rate-limit.ts` rebucketed: authenticated requests key primarily on `user:${userId}` (one abuser can no longer exhaust the bucket for every legitimate user), with a secondary `ip:${ip}` check when `perUser` is enabled and an IP is known. Anonymous requests key on IP only; a new `resolveIp()` helper honours `X-Forwarded-For` iff `TRUST_PROXY=true` is set in the env — never otherwise, since an untrusted client could otherwise spoof its bucket. When no IP is resolvable (e.g. unix-socket transport), we now fail-open with a `log.warn(namespace: 'rate-limit', …)` rather than collapsing all such traffic into a shared `'unknown'` bucket that itself amplifies the DoS.
- [x] **H7. `GET /api/runners/:runnerId` leaks cross-tenant** — `packages/server/src/routes/runners.ts:126-131`. Fix: check `runner.userId === userId` (admin bypass allowed). **Done**: `GET /:runnerId` now returns 404 (not 403) unless the caller is the runner's owner, an admin, or a runner-authenticated (server-to-runner) request — the 404 path prevents existence disclosure to other tenants. Ownership is resolved via `rm.getRunnerUserId(runnerId)` (the `RunnerInfo` shape intentionally hides `userId`, so the raw DB column is fetched on demand). Extended the same guard to the project-assignment endpoints — `POST /:runnerId/projects`, `GET /:runnerId/projects`, `DELETE /:runnerId/projects/:projectId` — via a shared `authorizeRunnerAccess(runnerId, c)` helper. Previously any authenticated user could read or mutate another tenant's runner-to-project bindings; now those attempts return 404 and leak no existence information.
- [x] **H8. Session `role` read via `as any`** — `packages/server/src/middleware/auth.ts:147` silently defaults to `'user'`. Fix: validate against a role enum; fail closed. **Done**: introduced `USER_ROLES` / `UserRole` / `normalizeUserRole()` in `packages/server/src/lib/types.ts` (the allowlist is `['admin', 'user']`). `ServerEnv.Variables.userRole` is now typed as `UserRole` instead of `string`. The auth middleware reads the session role through `normalizeUserRole()`, which coerces unknown values (or non-string) to the least-privileged `'user'` and logs a `namespace: 'auth'` warning when it had to coerce — so a crafted session or Better Auth schema drift can no longer pass an unexpected role through to downstream checks.
- [x] **H9. Admin first-run prints password to stderr** — `packages/server/src/lib/auth.ts:220-232`. Stderr is captured by log pipelines. Fix: force one-time password-setup token on first login. **Done**: the generated admin password is no longer written to stderr or through the logger. It is persisted to `~/.funny/admin-password.txt` with mode `0600` (owner-read-only), and stderr receives only the *file path* plus a "delete after first login" hint. Log pipelines (Abbacchio, journald, docker logs) that capture stderr therefore cannot harvest the credential. Operators that prefer not to create the file can continue to set `ADMIN_PASSWORD` in the environment to skip generation entirely.
- [x] **H10. Invite-link use-count increment not atomic** — `packages/server/src/routes/invite-links.ts:203-275`. Concurrent registrations can exceed `maxUses`. Fix: wrap create/add/increment in a DB transaction. **Done**: new `reserveInviteSlot(linkId)` helper issues a single conditional `UPDATE invite_links SET use_count = use_count + 1 WHERE id = ? AND revoked = '0' AND (expires_at IS NULL OR expires_at > now) AND (max_uses IS NULL OR use_count < max_uses) RETURNING id`. Both `POST /register` and `POST /accept` call it *before* creating/adding the user and bail with 410 when the slot can't be reserved. Because the check-and-increment lives in one SQL statement, two concurrent callers can no longer both pass the pre-check and both commit — the database now enforces the `max_uses` ceiling regardless of interleaving. The prior non-atomic `db.update(... set: { useCount: String(useCount + 1) })` calls (which did last-write-wins and could lose increments entirely) are removed. Works across SQLite and PostgreSQL via explicit `CAST(... AS INTEGER)` since the column is TEXT for dialect parity.
- [x] **H11. Client uses `containerUrl` unvalidated** — `packages/client/src/hooks/use-ws.ts:587-593`, `lib/api.ts:80-82`. Fix: validate against allowlisted origins. **Done**: added `validateContainerUrl(raw)` in `packages/client/src/lib/api.ts` — rejects non-`http(s)` schemes (blocks `javascript:`, `data:`, `file:`), URL-embedded credentials, malformed URLs, and (when `VITE_ALLOWED_CONTAINER_ORIGINS` is set) any origin outside that comma-separated allowlist. Returns the canonical `https://host:port` origin so callers cannot concatenate attacker-chosen path fragments. `getBaseUrlForThread()` in `api.ts` and `connectRemoteWS()` in `hooks/use-ws.ts` now validate through this helper and fall back to the same-origin API base (or log-and-refuse in the WS path) instead of blindly following a malicious `containerUrl` that a compromised thread record could set.
- [x] **H12. Missing authz on `browse/git-init`, `open-in-editor`, `open-terminal`** — `packages/runtime/src/routes/browse.ts`. Fix: confine to caller's registered projects (ties into C5). **Done**: resolved as part of the C5 path-scope rewrite. `POST /git-init` now runs through `requirePickerPath()` — scoped to `$HOME` (or a drive root on Windows), with credential dirs (`.ssh`, `.aws`, `.gnupg`, `.kube`, `.config/gcloud`, `.docker`) and a defensive `/etc|/proc|/sys|/dev|/run|/boot|/root|/var` blocklist rejected; the picker scope is required because git-init runs during project creation before any project record exists. `POST /open-in-editor` and `POST /open-terminal` now call `requireProjectPath(dirPath, userId)` which forces the path to sit under one of the caller's `listProjects(userId)` entries (or its worktree base) with `path + sep` containment to prevent sibling-prefix escapes — so a caller can't spawn a terminal or editor at another tenant's directory or at `/etc`, `/root`, etc. Rejections log `namespace: 'browse'`.

- [x] **M1. Mermaid SVG rendered via `dangerouslySetInnerHTML`** — `packages/client/src/components/MermaidBlock.tsx:62,326`. Fix: sanitize via DOMPurify SVG profile or render in sandboxed iframe. **Done**: added `sanitizeMermaidSvg(svg)` in `MermaidBlock.tsx` that runs the mermaid-rendered output through `DOMPurify.sanitize(..., { USE_PROFILES: { svg: true, svgFilters: true }, FORBID_TAGS: ['script', 'foreignObject'] })` before the string is assigned via `dangerouslySetInnerHTML`. Applied in the `useMermaidSvg` hook for the inline block and in the lightbox render path, so both surfaces are sanitized even if mermaid's `securityLevel: 'strict'` regresses or a future theme extension injects raw markup. `<script>` and `<foreignObject>` (the two standard SVG-XSS injection points for HTML-in-SVG) are dropped regardless of what mermaid emits.
- [x] **M2. Sidebar cookie missing `Secure`/`SameSite`** — `packages/client/src/components/ui/sidebar.tsx:108`. **Done**: `setOpen` now writes `document.cookie = \`${SIDEBAR_COOKIE_NAME}=${openState}; path=/; max-age=${SIDEBAR_COOKIE_MAX_AGE}; SameSite=Lax${secureAttr}\`` where `secureAttr = '; Secure'` iff `window.location.protocol === 'https:'`. `SameSite=Lax` blocks the cookie from being sent on cross-origin POST/sub-resource requests (the CSRF vector for a state cookie), and `Secure` is added only on HTTPS origins so it doesn't break the localhost dev HTTP flow. No server-side cookie was present — this is the only writer.
- [x] **M3. Runner namespace uses CORS origin check meant for browsers** — `services/socketio.ts:71-74`. Fix: enforce token-auth timeout on WS upgrade; drop origin gate for runners. **Done**: runner namespace middleware now races `rm.authenticateRunner(token)` against a `AUTH_TIMEOUT_MS = 10_000` timeout via `Promise.race`, returning the deterministic error `Authentication timed out` to the client socket instead of hanging the upgrade indefinitely (which previously left half-open sockets consuming server memory). The origin allowlist gate is kept on the browser `/` namespace (where it actually defends against cross-origin WebSocket attacks) but never applied to `/runner`, where runners legitimately connect from arbitrary hostnames/NATs and the only trust anchor is the shared-secret bearer token.
- [x] **M4. `runner-resolver` can return just-disconnected runner** — `packages/server/src/services/runner-resolver.ts`. Fix: invalidate on disconnect event. **Done**: the socket.io runner-disconnect handler now calls `resolver.evictRunnerFromCache(runnerId)` immediately after `wsRelay.removeRunnerClient(runnerId, socket.id)`, so the in-memory "online runner" cache used by `findRunnerForProject` / `findAnyRunnerForUser` drops the stale entry before the next request tries to route to it. Without this, a request arriving in the ~millisecond window between socket disconnect and the next DB poll would be routed to a dead socket and fail with a 502 after its own timeout (and, prior to C9, could have fallen through to another runner).
- [x] **M5. `directHttpFetch` forwards runner's raw response headers** — `packages/server/src/middleware/proxy.ts:184-188`. Malicious runner can set `Set-Cookie`, `Access-Control-*`, etc. on the server origin. Fix: safelist forwarded headers. **Done**: introduced `SAFE_RUNNER_RESPONSE_HEADERS` (`content-type`, `content-length`, `content-encoding`, `content-disposition`, `content-language`, `cache-control`, `etag`, `last-modified`, `vary`, `x-content-type-options`) and `filterSafeRunnerResponseHeaders(source: Headers): Headers` in `proxy.ts`. The `new Response(..., { headers: runnerResponse.headers })` return now uses `filterSafeRunnerResponseHeaders(runnerResponse.headers)` instead, so a compromised runner can no longer inject `Set-Cookie` (session fixation on the server origin), `Access-Control-Allow-*` (break CORS on the server's real origin), `Strict-Transport-Security`, `X-Frame-Options` overrides, or arbitrary custom headers that downstream middleware or CDNs might react to.
- [x] **M6. `ansi-to-html` XSS hinges on `escapeXML: true`** in multiple components. Fix: wrap in a single utility with a test asserting `<` escapes. **Done**: new `packages/client/src/lib/ansi-to-html.ts` exports `createAnsiConverter(options?: AnsiConverterOptions)` where `AnsiConverterOptions = Omit<ConstructorParameters<typeof AnsiToHtml>[0], 'escapeXML'>` — the type system *prevents* callers from passing `escapeXML` at all, and the factory hardcodes `{ ...options, escapeXML: true }`. All seven historical call sites (`ToolCallCard.tsx`, `BashCard.tsx`, `TerminalPanel.tsx`, `TestDetailTabs.tsx`, `SystemSettings.tsx`, `WorkflowEventCard.tsx`, `PipelineEventCard.tsx`) now import `createAnsiConverter` from `@/lib/ansi-to-html` instead of `AnsiToHtml` directly. Vitest suite `packages/client/src/__tests__/lib/ansi-to-html.test.ts` asserts: (a) `<` is always escaped in output, (b) it's *still* escaped when a caller spreads in an object that explicitly sets `escapeXML: false` (with `@ts-expect-error` to prove the type gate also fires), (c) ANSI colour codes still render as HTML spans. All three tests pass. Net effect: no future component can regress to unescaped-XML output by forgetting the option.
- [x] **M7. Directory-name validation allows `.`, `..`, Windows-reserved names** — `routes/browse.ts:186-190`. **Done**: `POST /create-directory` in `packages/runtime/src/routes/browse.ts` now rejects: bare `.` and `..`, names ending in `.` or space (both invalid on Windows and silently coerced), and the Win32 reserved device-name set `/^(CON|PRN|AUX|NUL|COM[1-9]|LPT[1-9])(\..*)?$/i` (including extensions — `CON.txt` resolves to the CON device on Windows just like `CON`). These are refused with HTTP 400 *before* the path-scope check so the operator gets a clear error rather than an opaque EPERM from Win32, and a cross-platform deployment that replicates to Windows can't be tricked into creating a directory the Windows filesystem layer will silently reinterpret.
- [x] **M8. Socket ack `requestId` not enforced unique** — `services/socketio.ts:543-555`. **Done**: `setupRunnerDataHandlers` now maintains a per-socket `const inFlightRequestIds = new Set<string>()` tracker. Before dispatching a `data:*` event, the handler checks if `requestId` is already in flight and — on a duplicate — emits `{ error: 'Duplicate requestId in flight', success: false }` on the expected `data:response:<requestId>` channel (so the original waiter still gets a deterministic response) and returns without starting a second handler. The Set is populated pre-dispatch and cleaned in a `finally` block covering both success and error paths, plus on `socket.on('disconnect', ...)`. Prevents two concurrent handlers from racing for the same ack channel (which previously could yield non-deterministic "last write wins" semantics and, worst-case, leak a stale resolver if the first handler's `finally` ran after the second handler had overwritten the entry).
- [x] **M9. Runner process model unsandboxed** — document that exposing a runner to a remote `TEAM_SERVER_URL` gives that server shell access to `$HOME`. **Done**: added a prominent trust-boundary callout to `INSTALL.md` in the **Machine B — Runner** setup section spelling out that a runner pointed at a remote `TEAM_SERVER_URL` effectively grants that central server shell execution in the runner's `$HOME` under the invoking OS user — including every repo that user can reach, saved GitHub tokens, provider API keys, and files under the `$HOME` picker scope (modulo the credential-dir blocklist: `.ssh`, `.aws`, `.gnupg`, `.kube`, `.config/gcloud`, `.docker`). The callout recommends hardening options: dedicated unprivileged OS user / VM / container per runner, `FUNNY_DATA_DIR` relocation, firewall scoping, and `RUNNER_AUTH_SECRET` rotation on decommission. Also appended a concise one-line bullet to the `## Security Notes` section pointing back to the full writeup, and added a matching warning plus link to `INSTALL.md` from the **Deployment Architecture** section of `CLAUDE.md` so anyone reading the project instructions (including agents working in this repo) sees the boundary before wiring team mode. This is documentation-only — the sandboxing itself is out of scope for the audit.

Use structured client logger. **Done**: `auth-store.ts` now creates `const log = createClientLogger('auth-store')` from `@/lib/client-logger` and replaces the bare `console.error('[auth-store] initialization error:', err)` with `log.error('initialization error', { error: err instanceof Error ? err.message : String(err) })`.
